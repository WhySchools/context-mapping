# context-gen Design Decisions

This document summarizes the important design decisions and invariants for context-gen.
It is meant as a fast onboarding document for humans and agents before changing the tool.

## Purpose

context-gen creates AST-based context maps for software projects.

Each indexed module gets a `.context/*.md` file with two zones:

- `[auto]`: generated from source code, kept in sync by the tool.
- `[manual]`: written by humans or explicitly guided agents, used to preserve intent.

The tool exists to reduce prompt/context size while preserving the reasoning that code cannot express: architecture decisions, invariants, test strategy, and deliberately unimplemented behavior.

## Core File Contract

The central contract is:

```text
AUTO_START ... AUTO_END     may be regenerated by the tool
MANUAL_START ... MANUAL_END must never be overwritten by the tool
```

Any change to `merger.py` must verify that the `[manual]` section is preserved.

Agents must not edit `[auto]` sections by hand. If generated output is wrong, fix the generator.

## Parser Registry

`cli.py` must not know which languages exist.

Language support is provided through `ParserPlugin` objects registered into `schema.REGISTRY` by parser modules:

```text
schema.REGISTRY
  - rust        registered by parsers/rust_parser.py
  - typescript  registered by parsers/ts_parser.py
  - php         registered by parsers/php_parser.py
```

Adding a language means:

1. Create `parsers/<lang>_parser.py`.
2. Register a `ParserPlugin` at the end of that file.
3. Add one import line in `cli.py` to trigger registration.

Do not add `if language == ...` dispatch to `cli.py`, `merger.py`, or `schema.py`.

## Directory Boundary

Directory parsers are intentionally non-recursive.

```text
find_dirs(root) discovers directories recursively.
parse_dir(dir, root) parses only files directly inside that directory.
```

This keeps parser behavior predictable and prevents one context file from absorbing unrelated submodules.

If a task asks for parser recursion, record a tension before implementing.

## Load Stdout Contract

`load` is intended for piping into clipboards, prompts, and other tools:

```bash
context-gen load <path> . --include-manual | <anything>
```

Its stdout must contain only context content.

Errors and operational noise must go to stderr. Any new code path used by `load` must preserve this.

## Language-Aware Rendering

`FunctionInfo.signature(language)` was introduced because rendering every function as Rust syntax misled readers when indexing TypeScript or PHP.

Current conventions:

- Rust: `pub async fn name(params) -> ReturnType`
- TypeScript: `export async function name(params): ReturnType`
- PHP: `public function name(params): ReturnType`

However, this is already a pressure point. Future language support should not casually add more `if language == ...` branches to core schema/rendering. If a new parser needs specialized signature rendering, propose a rendering design first.

## IPC / Entry Point Labels

The meaning of an "exposed function" depends on the ecosystem.

Examples:

- Rust/Tauri: `#[tauri::command]`
- WordPress/PHP: `add_action()` and `add_filter()`
- TypeScript: exported API surface

This label belongs to `ParserPlugin.ipc_label`, not to hardcoded renderer text.

Parser plugins own ecosystem-specific semantics.

## WordPress/PHP Scope

The PHP parser treats WordPress hooks as the PHP plugin equivalent of an IPC bridge.

Accepted V1 limitation:

- Detect string literal callbacks such as `add_action('init', 'my_callback')`.
- Do not resolve array callbacks such as `[$this, 'method']`.
- Do not resolve closures.

False negatives are preferred over false positives because agents trust generated context.

`vendor/` must remain skipped for PHP projects.

## Global Context

`.context/GLOBAL.md` is the entrypoint context file.

It summarizes:

- detected tech stack
- module index
- dependency highlights when available

Important decisions:

- Tech stack detection must reflect actual files, not assume Rust by default.
- PHP projects should be detected without scanning skipped third-party directories.
- Module index links must use the same path-to-context naming rule across Windows and Unix path separators.
- `generate_global_context(..., quiet=True)` must avoid operational stdout noise.

## Prompt-Assisted Parser Creation

New parsers must use a proposal-first workflow.

Before implementation, use:

```text
docs/prompts/add-parser.md
```

The proposal must cover:

- package availability
- AST probe plan
- extraction scope
- IPC/entrypoint equivalent
- files expected to change
- tests
- tensions

Human approval is required before code changes.

## Tension Register

When task requirements conflict with project invariants, agents must write to `.context/TENSIONS.md` before continuing.

Low severity tensions may continue conservatively.
High severity tensions must stop and wait for a human decision.

Common tension triggers:

- adding language-specific dispatch to core files
- printing noise to `load` stdout
- overwriting `[manual]`
- making parsers recursively scan directories
- expanding scope beyond documented unimplemented behavior

## Current Known Gaps

These are known gaps, not accidental omissions:

- Python parser is not implemented.
- Go parser is not implemented.
- PyPI packaging is not implemented.
- `[manual]` staleness detection is not implemented.
- Multi-language directory behavior is not fully defined.
- Signature rendering may need a plugin-owned design before more languages are added.

## Operational Notes

On Windows, prefer Debian WSL for running context-gen.

The current local setup exposes:

```bash
context-gen
```

through:

```text
~/.local/bin/context-gen
```

That wrapper activates:

```text
~/.venvs/context-mapping
```

and runs:

```text
/mnt/d/Github/context-mapping/cli.py
```

This is an operational convenience, not a packaging substitute. Proper package installation still requires future `pyproject.toml` work.
